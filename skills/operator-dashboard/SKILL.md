# Operator Dashboard

## Overview

Use this skill to maintain a clean operator-facing view of system state.

The purpose is to make current work visible enough that Dylan can understand what matters without having to reconstruct everything from scattered conversations and files.

## Update Triggers

Use this skill when:

- a project starts
- a project changes stage
- a blocker appears or clears
- a priority changes
- an item moves from active to waiting or complete
- the dashboard feels stale or misleading
- Zee wants to improve visibility into current operations

## Dashboard Categories

A useful dashboard should normally track:

- current focus
- active priorities
- blockers
- waiting items
- recent wins
- next actions

Add more only if they improve clarity.

## Priority Rules

Priorities should reflect:

- what most directly serves Dylan's current goals
- what is time-sensitive
- what is blocked vs movable
- what creates leverage for future work
- what should **not** be allowed to disappear into background noise

Do not list everything as high priority.

## Status Definitions

Use clear state distinctions such as:

- active
- blocked
- waiting
- paused
- complete

Choose the narrowest accurate state.

Do not label something active if it is really waiting. Do not label something blocked if it merely needs effort.

## Waiting-On And Blocker Tracking

Track:

- what is waiting
- who or what it is waiting on
- whether the wait is normal or concerning
- what would unblock it

This matters because hidden waiting items create confusion and make the system feel less trustworthy.

## Review Cadence

Review and refresh the dashboard when:

- meaningful work happens
- priorities change
- a blocker emerges or clears
- the current view no longer matches reality

Do not let the dashboard become a stale decorative file.

## Escalation Thresholds

Surface things more clearly when:

- an important task has been waiting too long
- a blocker is real and persistent
- the current priority is not moving
- multiple active efforts are competing and need a decision
- Dylan would reasonably want to know the state now

## Summary Output Format

When updating or presenting the dashboard, prefer a compact structure like:

- current focus
- active priorities
- blockers
- waiting items
- recent wins
- next actions

The view should be concise but informative.

## Freshness Rule

A dashboard loses value when it is outdated.

Remove or update:

- completed items that still look active
- stale blockers that are no longer real
- old priorities that no longer matter
- waiting items with no useful explanation

Freshness is part of accuracy.

## Guiding Principle

Use this rule:

> Show the real operating state clearly enough that Dylan can understand what matters now.

And this counter-rule:

> Do not turn the dashboard into a giant archive that hides the important things under too much status clutter.

## Intent

This skill exists to make Zee's work more visible, legible, and trustworthy.

The goal is:

- clearer current state
- fewer hidden blockers
- better priority visibility
- easier oversight without micromanagement
