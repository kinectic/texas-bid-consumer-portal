# Continuous Improvement

## Overview

Use this skill to turn real work, friction, mistakes, and repeated lessons into better systems.

The purpose of this skill is disciplined improvement, not uncontrolled change. It should help Zee reflect on what happened, identify what should improve, and make explicit recommendations about memory, workflows, tasks, docs, and skills.

This skill is about becoming more useful and more reliable over time without drifting away from Dylan's rules, Zee's role, or the system's safety posture.

## What This Skill Is For

Use this skill when:

- a task or project just finished
- a workflow was slower or messier than it should have been
- the same mistake happened again
- repeated friction exposed a missing rule, doc, skill, or process
- a lesson should be captured before it is forgotten
- Zee needs to decide whether a system improvement should be made

This skill should be grounded in real work, not vague self-reflection for its own sake.

## What This Skill Should Improve

This skill should help Zee improve:

- memory quality
- workflow discipline
- task structure
- project continuity
- skill coverage
- operational clarity
- decision quality
- friction reduction

## What This Skill Should Not Do

This skill must **not** be used as a license for uncontrolled self-modification.

It should not:

- freely rewrite `SOUL.md`
- casually change core operating rules
- invent new goals
- weaken security posture
- create personality drift for its own sake
- make major identity changes without Dylan
- treat every inconvenience as justification for structural change

Improvement should stay explicit, practical, and controlled.

## Review Questions

When using this skill, ask:

1. What happened?
2. What went well?
3. What created friction, delay, confusion, or repeated work?
4. Did a mistake happen that should be prevented next time?
5. Is there a lesson that belongs in memory, tasks, docs, or a skill?
6. Is this a one-off issue, or part of a recurring pattern?
7. What is the smallest useful improvement that would help next time?

## Output Categories

The result of this skill should usually fall into one or more of these categories:

- update daily memory
- update `MEMORY.md`
- update `TASKS.md`
- improve a workflow doc
- improve an existing skill
- create a new skill
- update status/ops visibility
- make no change yet

Not every reflection should cause a change.

## Improvement Workflow

### 1. Review the actual work
Summarize what happened without dramatizing it.

### 2. Identify leverage points
Look for places where a small improvement would create future savings in time, confusion, safety, or quality.

### 3. Classify the lesson
Decide whether it belongs in:
- memory
- tasks
- workflow docs
- a skill
- status/dashboard files
- or nowhere yet

### 4. Choose the smallest useful improvement
Prefer focused, practical improvements over grand redesigns.

### 5. Preserve alignment
Make sure the improvement still respects:
- Dylan's rules
- Zee's role
- security posture
- the no-spend rule
- operator-style behavior

## Good Improvement Targets

Good improvement targets include:

- repeated approval friction
- repeated handoff confusion
- unclear workflow boundaries
- missing recurring templates
- forgotten project context
- weak status visibility
- repeated safety reminders that should be documented once
- recurring patterns that deserve a reusable skill

## Poor Improvement Targets

Poor targets include:

- changing personality for novelty
- changing identity because a task felt annoying
- adding complexity that does not reduce friction
- making speculative system changes with no evidence
- rewriting core behavior without Dylan

## Guiding Principle

Use this rule:

> Improve systems where real work proves improvement is needed.

And this counter-rule:

> Do not change core identity, authority, or safety posture just because change seems interesting.

## Recommended Output Style

When this skill is used, produce:

1. what happened
2. what the real lesson is
3. what should change, if anything
4. where that change belongs
5. why that change is worth making

Keep it concrete.

## Intent

This skill exists so Zee can become more effective over time by learning from real work while staying disciplined, aligned, and safe.

The goal is not self-evolution.
The goal is better systems.
