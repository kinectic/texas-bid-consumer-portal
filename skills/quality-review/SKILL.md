# Quality Review

## Overview

Use this skill to decide whether work is actually ready.

The purpose is to prevent weak, incomplete, inaccurate, unsafe, or sloppy outputs from being treated as finished just because effort has already been spent.

## Review Triggers

Use this skill when:

- a task appears complete
- a draft may be sent outward
- a batch may be treated as ready
- a summary, file, or result may be relied on
- multiple helper outputs must be checked before acceptance
- the cost of being wrong is higher than the cost of a short review

## Review Dimensions

Review across these dimensions as applicable:

- accuracy
- completeness
- clarity
- safety
- tone
- formatting
- alignment with the stated goal
- readiness for external use

Not every dimension matters equally every time. Use judgment.

## Risk-Based Depth Rule

Review depth should match the risk.

### Light review
Use when the result is low-risk, internal, and easily corrected.

Check:
- obvious correctness
- obvious completeness
- obvious formatting issues

### Standard review
Use when the work matters but is still internal or reversible.

Check:
- goal alignment
- key facts
- completeness of the result
- clarity and usefulness
- whether anything obvious is missing

### Strict review
Use when the result may go outward, shape decisions, or create downstream consequences.

Check:
- factual accuracy
- completeness against the goal
- consistency across sections
- safety and approval boundaries
- tone/channel fit
- whether the result is truly ready or only looks finished

## Verification Checklist

Before passing work, ask:

1. Did this actually answer the task?
2. Is anything important missing?
3. Is anything likely wrong, weak, or unverified?
4. Is the result clear enough to use?
5. Does the tone fit the context?
6. Does anything create avoidable risk?
7. Would Dylan be surprised or disappointed by the current quality level?

## External-Send Readiness Check

If the output might go outward, also check:

- is the message accurate?
- is the ask clear?
- is the tone appropriate?
- does it expose anything it should not?
- does it require explicit approval before sending?
- does it still need personalization or cleanup?

If the answer is no, it is not ready.

## Defect Classification

Classify problems as:

- **fix now** — clear issue that should be corrected before acceptance
- **flag** — not ideal, but acceptable if surfaced clearly
- **accept** — good enough for the current risk/context

Do not pretend weak work is strong just to move faster.

## Signoff Rule

When review is complete, return one of these outcomes:

- pass
- revise
- pass with caveats
- not ready

Include:

- what was reviewed
- the main issues found, if any
- whether the result is ready
- what must change if it is not ready

## Pass vs Revise Guidance

Pass when:

- the result is accurate enough
- complete enough
- clear enough
- safe enough
- and aligned with the goal

Revise when:

- core facts look shaky
- the result is missing meaningful pieces
- the output is sloppy enough to hurt trust
- the tone or safety posture is wrong
- the work only looks complete at a glance

## Guiding Principle

Use this rule:

> Do not confuse finished-looking work with ready work.

And this counter-rule:

> Do not over-review low-risk internal work so heavily that quality control becomes paralysis.

## Intent

This skill exists to create a clean quality gate between execution and acceptance.

The goal is:

- fewer sloppy outputs
- fewer avoidable mistakes
- more trustworthy results
- better readiness judgment
