# Skill Development

## Overview

Use this skill to decide when a new skill should exist, capture the idea before it gets lost, and keep the skill system disciplined instead of bloated.

The goal is not to create as many skills as possible. The goal is to recognize when a workflow has become skill-worthy, preserve it cleanly, and avoid turning one-off tasks into clutter.

## Recognition Triggers

A new skill is probably needed when one or more of these are true:

- the same workflow is repeated multiple times
- the same instructions or cautions keep getting repeated
- a task has special rules, edge cases, or security concerns
- a tool or process is fragile and needs a repeatable playbook
- useful knowledge keeps living only in chat instead of in a reusable place
- a project becomes a recurring operating lane
- Zee explicitly thinks or says that something should probably become a skill

## What Counts As Skill-Worthy

A skill is a good fit when the pattern:

- will likely happen again
- has enough structure to document cleanly
- would save future Zee real time, confusion, or mistakes
- benefits from a repeatable workflow
- has reusable guidance, rules, scripts, or references

Good candidates include:

- repeated workflows
- repeated formatting or output rules
- recurring safety checks
- recurring outreach, research, operations, or setup playbooks
- domain-specific processes with non-obvious judgment rules

## What Does Not Count

Do **not** make a skill just because something happened once.

A new skill is usually **not** needed when the item is:

- a one-off task
- too vague to define clearly
- temporary and unstable
- better stored as a note, memory rule, or task entry
- just a project status update
- not reusable enough to justify a skill

Use `MEMORY.md`, daily memory, `TASKS.md`, or a project note instead when that is the better fit.

## Evaluation Checklist

Before creating or upgrading a skill, check:

1. Is this a real recurring need?
2. Is it stable enough to document?
3. Would a future Zee benefit from having this as a reusable playbook?
4. Is it better as a skill than as memory, a task template, or a project note?
5. Does it have clear triggers for when the skill should be used?
6. Can the workflow be described without excessive ambiguity?

If the answer is mostly no, do not create the skill.

## Capture Workflow

When a possible new skill appears:

1. Capture the trigger that made it feel skill-worthy.
2. Save example use cases or representative requests.
3. Save why the pattern matters.
4. Save what reusable pieces may be needed:
   - workflow steps
   - references
   - scripts
   - assets
5. Save whether the pattern should become:
   - a new skill
   - an update to an existing skill
   - a memory rule
   - a task template
   - or just a note

Do not trust conversational memory alone. Write it down.

## Upgrade Workflow

If the pattern is worth keeping as a skill:

1. Define the purpose of the skill.
2. Define the trigger conditions for when to use it.
3. Define the workflow or guidance it should contain.
4. Identify what bundled resources it needs, if any.
5. Keep the skill as small and focused as practical.
6. Avoid overbuilding it before real usage proves the need.

The skill should exist to create leverage, not ceremony.

## Maintenance Rule

A good skill system stays disciplined.

That means:

- create skills when they provide real leverage
- improve skills when real use exposes gaps
- avoid making skills for temporary or weak patterns
- clean up or consolidate stale skills when they no longer earn their keep

## Default Operating Rule

Use this rule as the baseline:

> If a workflow repeats, has special rules, or would save future Zee real time or confusion, evaluate it for skill creation.

And this counter-rule:

> If it is one-off, unstable, or better handled as memory or a note, do not make it a skill.

## Output Guidance

When this skill is used, the result should usually be one of these:

- **Create a new skill**
- **Update an existing skill**
- **Capture as memory instead**
- **Capture as a task/template instead**
- **Do nothing yet**

Keep the judgment practical. The purpose is better reuse, not more files.
