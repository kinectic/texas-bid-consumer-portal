# CONTINUITY-PROTOCOL.md

Rules for preserving important conversation context so Zee and Dylan can continue work across sessions without relying on raw chat history.

## Core Principle

Chat is temporary.
Files are real memory.

Do not assume an important conversation point will remain safely available in raw chat history.
If it matters later, write it down.

## Continuity Structure

### 1. Chat
Use for:
- live conversation
- quick decisions
- coordination
- temporary working context

Do not treat chat as durable memory.

### 2. Daily Memory
Use `memory/YYYY-MM-DD.md` for:
- what mattered that day
- project progress
- notable decisions
- lessons learned
- important instructions
- temporary but meaningful operating context

Rule:
- if it would matter tomorrow, it should probably be written to daily memory today

### 3. Long-Term Memory
Use `MEMORY.md` for:
- stable rules
- standing preferences
- long-term decisions
- security rules
- identity/role expectations
- durable workflow rules

Rule:
- if it changes how Zee should operate in the future, it should not live only in chat

### 4. Project Files
Use project/workspace files for:
- structured project state
- batch status
- playbooks
- lists
- campaign data
- working documents

Rule:
- if the information belongs to a project, put it in the project file instead of leaving it only in memory or chat

### 5. Skills
Use skills for:
- reusable workflows
- repeatable judgment patterns
- disciplined operating methods
- high-value recurring procedures

Rule:
- if the pattern is reusable and would help future Zee repeatedly, consider making or updating a skill

## Capture Rule

When something important appears in conversation, decide where it belongs:

- daily memory
- long-term memory
- a project file
- a skill
- nowhere yet

Do not leave important things floating in chat by default.

## Trigger Rule

Capture conversation points when they are:

- important tomorrow
- important next week
- likely to matter in a future session
- operationally important
- safety-relevant
- project-relevant
- workflow-changing
- a meaningful lesson

## Minimum Safe Rule

Use this baseline:

> If a conversation point would matter tomorrow, write it to a file today.

And:

> If it changes how Zee should operate in the future, it must not remain only in chat.

## Intent

This protocol exists so conversation continuity does not depend on raw session history surviving unchanged.

The goal is:
- fewer lost decisions
- better continuity
- stronger long-term usefulness
- safer memory handling
- less dependence on chat compression
